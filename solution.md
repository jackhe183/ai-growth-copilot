agent loop的核心机制就是异步请求大模型api，await里塞入多个工具调用，时间消耗等于耗时最长的那一个。deepseek网页版就是最好的agent loop，有完善的工具调用和大模型思考链路，以及不影响上下文效果的记忆压缩。

vibecoding搭建了react-agent：
```
PS C:\Users\jackHe\Desktop\react-agent> bun run src/index.ts "现在几点了？"                                              
任务: 现在几点了？

正在启动 ReAct 循环...
```

可以看到只有输入和输出，并没有展现出这个过程，类似于deepseek网页版那样的可溯源的机制，需要加入trace机制来观察内部。

**async def**：异步函数，执行这个函数产生的空闲时间可以去干别的
**await**：到了这里必须停下，做完了再走，例如
```
const response = await fetch(API_URL, {...});
```

技术选型：
之所以选择react范式，是因为要agent拥有执行工具、工具调用能力
之所以要调用工具来查数据，是因为要agent忠实于事实，减少幻觉发生可能性
提示词里加上动作边界描述，明确指出哪些是不能做的

业务闭环MVP以及责任边界，agent背后需要有人来绑定责任，agent不能做一件自动发邮件，员工的agent需要对客户负责，

首先是梳理出业务的SOP：

寻找线索数据（这个环节已经在mock数据里面有了，在实际过程中获取精准的客户线索数据也是不少的工作量）

联想了一下例如企业抖音账号底下有评论，需要找到联系方式和客户需求来持续跟进，了解清楚价格和服务配置，双方进一步确定商务合作，并达成合作合同，进而交给后面的交付实现。

这个营销agent的责任边界就在于从获客到销售之间，营销的职责是交付“线索”，agent需要辅助判断，客户“线索”是否及格，“线索”是营销和销售之间重要的传递接力棒的中间产物。

沉淀SOP为Agent使用的skill

agent按 Reasoning + Acting + Observation 来执行skill，输入trace可溯源，方便开发调试和可信溯源
