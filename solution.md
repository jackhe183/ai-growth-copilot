## 业务侧

### 思考过程

业务闭环MVP以及责任边界，agent背后需要有人来绑定责任，agent不能做一件自动发邮件，员工的agent需要对客户负责，

首先是梳理出业务的SOP：

寻找线索数据（这个环节已经在mock数据里面有了，在实际过程中获取精准的客户线索数据也是不少的工作量）

联想了一下例如企业抖音账号底下有评论，需要找到联系方式和客户需求来持续跟进，了解清楚价格和服务配置，双方进一步确定商务合作，并达成合作合同，进而交给后面的交付实现。

这个营销agent的责任边界就在于从获客到销售之间，营销的职责是交付“线索”，agent需要辅助判断，客户“线索”是否及格，“线索”是营销和销售之间重要的传递接力棒的中间产物。

沉淀SOP为Agent使用的skill，agent按 Reasoning + Acting + Observation 来执行skill，输入trace可溯源，方便开发调试和可信溯源

agent测评需要结合业务的实际需求偏好来制作测评集，收敛agent能力。例如在某些场景下的营销工作里，到底该要把哪些重复工作给ai自动化掉，哪些东西是agent很好做但是人做起来很麻烦的事情。

### 人在业务的位置

B2B 的流程大概是这样的：
```
市场/营销团队                    销售团队
    │                               │
    │  获客（广告、活动、SEO）        │
    │  收集线索（表单、私信）         │
    │  初步筛选线索                  │
    │  ──── 交接合格线索 ────────────▶│
    │                               │  跟进（打电话、发邮件）
    │                               │  摸清预算/决策人/时间节点
    │                               │  演示 Demo
    │                               │  报价谈判
    │                               │  签合同
```

营销的 KPI 是"交多少合格线索给销售"，销售的 KPI 是"转化多少单"。两者矛盾点在于：营销交的线索烂 → 销售抱怨浪费时间 → 两个团队互相甩锅。

agent 要解决的就是这个摩擦点：帮营销团队在交接之前先把线索过一遍，判断值不值得交给销售。


### agent在业务的位置边界

agent实际上需要有责任绑定机制，例如阿里里面一个子agent做的简单工作就是对应一个人的职责，这样子把部门里面的各个人职责内化成agent，实现数字资产，员工对agent负责，也是对公司业务负责。

agent能做的：查找文档、聚合展示、数据分析、辅助给人提效重复性工作
agent不能做的：替代人的业务判断、直接回复客户、多轮诊断分析

### agent测评结合业务期望产出来做

agent测评量化agent落地能力效果，构建多轮对话测评体系，角色模拟对话收集数据，方便快速迭代效果。测评集根据真实业务数据进行整理清洗，构建以首轮问题为锚定的测评数据集，忠实于真实业务场景数据，为客户使用效果负责。加入必要的数据pipeline。

claudecode本身就是很好的agent，参照claudecode的架构来设计快速迭代agent
